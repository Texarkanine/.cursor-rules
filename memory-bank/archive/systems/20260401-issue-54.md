# TASK ARCHIVE: Fix L4 Sub-Run Completion Flow — Rework: Option B

## SUMMARY

Fixed the Level 4 sub-run completion flow so that sub-runs now re-enter through `/niko` instead of `/niko-archive`, with proper cleanup of sub-run state and correct detection of standalone Level 1 completion. Then refactored the design so that `/niko` owns all state routing while `complexity-analysis.mdc` is a pure classifier, reducing complexity and clarifying responsibilities.

## REQUIREMENTS

- Ensure L4 sub-runs no longer jump directly to `/niko-archive`, but instead re-enter via `/niko` with the correct milestone and cleanup behavior.
- Detect task completion consistently using reflection status (`REFLECT COMPLETE`) or Level 1 + QA pass, so L4 and standalone workflows share the same completion criteria.
- Separate concerns so that `/niko` handles state routing and milestone transitions, while `complexity-analysis.mdc` focuses solely on classification.

## IMPLEMENTATION

- Updated the L4 re-entry behavior so that completed sub-runs return to `/niko` for state routing, rather than going straight to `/niko-archive`, aligning the flow with the intended entry point.
- Moved all state routing logic (L4 milestone transitions, sub-run resume behavior, existing-work checks, rework vs fresh routing) into `/niko` and expressed the branching via a mermaid state diagram, with prose only describing node bodies.
- Simplified `complexity-analysis.mdc` by removing the re-entry check and replacing it with a small "classification target" step, shrinking the file substantially and making it focused on deciding what kind of work is being requested.
- Updated references (including `milestones.mdc` and Level 4 workflow docs) so they now point at `/niko` as the state router instead of embedding routing assumptions in complexity analysis.

## TESTING

- Verified that both the original build and the rework build produced clean diffs and coherent workflows.
- Used preflight to catch missing `milestones.mdc` cross-references introduced by the refactor and corrected them before finalizing the design.
- Used QA to surface a pre-existing edge case (first entry after L4 plan) and fixed it in-line while preserving the new separation of concerns.
- Manually exercised the L4 sub-run completion and re-entry paths to confirm that sub-runs cleanly hand control back to `/niko` and that existing work is detected correctly.

## LESSONS LEARNED

- Separating state routing from classification makes both `/niko` and `complexity-analysis.mdc` easier to understand and maintain; routing and classification are distinct concerns and should live in different places.
- Reusing existing signals (reflection status and Level 1 + QA pass) to infer "task complete" is better than introducing new markers or flags, and works consistently across L4 and standalone flows.
- The diagram-carries-branching pattern scales well even as `/niko` absorbs more logic: the mermaid diagram can hold complex state transitions while prose stays readable and focused on node behavior.

## PROCESS IMPROVEMENTS

- Allowing an explicit rework pass after QA and reflection, but before archive, is a good seam for architectural improvements that do not change behavior.
- Preflight and QA together provide strong coverage: preflight is well-suited to catching missing dependencies and cross-references, while QA is better at surfacing behavioral edge cases that emerge from refactors.

## TECHNICAL IMPROVEMENTS

- Future workflow changes should treat `/niko` as the single state router and keep classification logic isolated, to avoid reintroducing conflated responsibilities.
- The widened "task complete" criteria should be treated as a shared concept and reused wherever completion is checked, to keep semantics aligned across levels.

## NEXT STEPS

- None. The L4 sub-run completion flow and state routing/classification separation are now stable, and the system is ready for future enhancements that build on this architecture.
